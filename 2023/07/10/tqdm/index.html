tqdm使用

是一个进度条包，通过在shell上不断重写来达到进度条的效果。

最常用的在于下载上，如果你用linux下过东西，就很熟悉这种形式了。

万物均可查文档，如有需要自行查看：

文档 （ctrl+左键，点开连接的操作，pdf里面也是一样）

第一种使用方式

第一种使用方式就是题上写的

for i in tqdm(range(1,100)):
    sleep(0.2)

这个i的类型？其实还是对range(1,100)迭代，所以是tqdm中的迭代参数的类型

它体现在输出上是：

27%|██▋       | 27/99 [00:05<00:14,  4.81it/s]

00:05<00:14

左边指已经进行了的时间，右边指还剩多少时间，后面是速度，it/s就是item/s

27/99

这个是已经进行了多少。

这个多少指的是：range(1,100)中的多少个。一个算作一item

每个item都会花0.2秒（sleep的功能，这个函数就是让啥指令也不执行，去停0.2s，学了多线程或者操作系统就知道了)

用for形式的必须要迭代！！！，不然不知道在循环啥

tqdm()这个里面必要的一个参数，就是一个可供迭代的对象，这样才能显示运行的进度

iterable=None,            
desc=None,      传入str类型，作为进度条标题（类似于说明）
total=None,     *预期*的迭代次数
leave=True,             
file=None, 
ncols=None,         可以自定义进度条的总长度
mininterval=0.1,    最小的更新间隔
maxinterval=10.0,   最大更新间隔miniters=None, ascii=None, unit='it',unit_scale=False, dynamic_ncols=False,

但是for+total会发生啥？

for i in tqdm(range(0,100),total=20):
    sleep(0.2)

例如这个

混用的话会出现：

刚开始

45%|████▌     | 9/20 [00:01<00:02,  4.70it/s]

可以看见，这个9/20是属于total的

进行完这个之后，会有

25it [00:05,  4.79it/s]

这个，到100的时候会停，看来是100的奏效了

所以不要混用。

不用total，我们来试着加一下其他参数

for i in tqdm(range(0,100),desc="download"):
    sleep(0.2)

这样的效果是：

download: 100%|██████████| 100/100 [00:20<00:00,  4.97it/s]

注意看，这里设置参数的时候，是让desc=” “。面对这种参数很多的，缺省的，一般都采用这种参数名+= 的操作来实现设置特定的参数，其他都是默认

for i in tqdm(range(0,100),desc="download",ncols=100):
    sleep(0.2)

换成这样呢？可以看到是控制进度条长度的

download:  21%|██████████▉                                         | 21/100 [00:04<00:15,  4.98it/s]

变长了捏

postfix

for i in tqdm(range(0, 100), desc="download", postfix="books"):
    sleep(0.02)

我们先使用一下这个

输出是：

download: 100%|██████████| 100/100 [00:02<00:00, 48.91it/s, books]

诶，好像仅仅就是一进度条的后缀

但是，它可以表征一个参数环境，例如

用一个字典的形式体现参数：

for i in tqdm(range(0, 100), desc="download", postfix={"os":"linux"}):
    sleep(0.02)

输出的时候可以体现参数环境：

download: 100%|██████████| 100/100 [00:02<00:00, 48.62it/s, os=linux]

后边那俩在for的时候是不能体现的。

比如我去掉sleep，实现一个很小的算数：

for i in tqdm(range(0, 100), desc="download", ncols=100, mininterval=None):
    a = 1 + 1

好家伙，你看看这速度

download: 100%|████████████████████████████████████████████████| 100/100 [00:00<00:00, 11760.61it/s]

但是，不想这么快，可能要和其他线程汇合？

你需要换用一种方式：

第二种使用方式

第二种是一种上下文的方式，方便对于这个东西进行更好的控制。

我们看：

for i in tqdm(range(0,100),desc="download",ncols=100):

tqdm应该是一个可以迭代的对象，所以其实所有可迭代的对象都可以作为迭代的东西传进来的。

比如

for i in tqdm(["a","b","c","d"], desc="download", postfix={"os":"linux"}):
    sleep(0.2)

其效果为：

download: 100%|██████████| 4/4 [00:00<00:00,  4.99it/s, os=linux]

但是这样的话，并不能体现到底是再哪个

我们可以：

from time import sleep

from tqdm import tqdm
x=["a","b","c","d"]
pbar= tqdm(x, desc="download:c", postfix={"os":"linux"})
for i in pbar:
    pbar.set_description(i)
    sleep(0.2)

可以及时修改desc的内容。

另外，方便自己更新进度条，我们可以选择使用with的模式：

with tqdm(x, desc="download:c", postfix={"os": "linux"}) as pbar:
    for i in pbar:
        sleep(0.2)
        pbar.set_description(i)

这个和上面的是等价的。

我们可以不可以不去使用for迭代呢？

这里介绍一个新的函数：update：

更改之后：

with tqdm(x, desc="download:c", postfix={"os": "linux"}) as pbar:
        sleep(0.2)
        pbar.update(1)

会卡到这里：

download:  25%|██▌       | 1/4 [00:00<00:00,  4.98it/s, os=linux]

说也正常，因为毕竟就update了一次，我可以更换一下：

x = ["a", "b", "c", "d"]
with tqdm(x, desc="download", postfix={"os": "linux"}) as pbar:
    for i in x:
        sleep(0.2)
        pbar.update(1)

注意，这样的情况下，我们是对x迭代，而不是对tqdm迭代。这个可以容许我干很多事。

这个update()里面的整数，就是这个方法执行的迭代次数，就是进度条里面a/b的那个a的变化。

上面说到，total，是预期的迭代次数，但是超出预期呢？

比如：

import time
from tqdm import tqdm

with tqdm(total=200) as pbar:
    pbar.set_description('Processing:')
    # total表示总的项目, 循环的次数20*10(每次更新数目) = 200(total)
    for i in range(20):
        # 进行动作, 这里是过0.1s
        time.sleep(0.1)
        # 进行进度更新, 这里设置10个
        pbar.update(10)
Processing:: 100%|██████████| 200/200 [00:02<00:00, 99.16it/s]

没毛病。

但是我修改一下：

with tqdm(total=200) as pbar:
    pbar.set_description('Processing:')
    # total表示总的项目, 循环的次数20*10(每次更新数目) = 200(total)
    for i in range(30):
        # 进行动作, 这里是过0.1s
        time.sleep(0.1)
        # 进行进度更新, 这里设置10个
        pbar.update(10)

那么，

Processing::  80%|████████  | 160/200 [00:01<00:00, 94.27it/s]

刚开始的时候还是只有200个，这个是预期的循环次数，预期的循环次数决定了一开始输出的/200，所有的80%的比例都是按照这个计算的

当这个200跑完后，会进入到这个状态：

Processing:: : 290it [00:03, 96.17it/s]

就是没有进度条的状态
