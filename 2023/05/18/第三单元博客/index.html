黑箱测试和白箱测试

黑箱测试

黑箱测试，是指不知道软件内部的构成，单纯投入数据然后根据输出的结果来判定软件的正确与否。

我们平时自己写的测评机就类似这种，生成数据，然后测评机根据这个数据进行计算并投喂给我们待测试的程序，依据程序的输出比较来判定执行的正确性。

事实上，由于采用了黑盒，程序的每一个部分对我们来说都是完全封闭的，并不能知道任何信息。因此，在出现漏洞的时候，程序是作为一个整体出现的，因此定位bug会显得困难。

并且，由于不知道会出什么问题，对整个程序的测试会涉及很多方面，整个的测试会将所有各个方面的数据投入给一个程序整体，这就导致了效率的低下。

同时，构造的数据也难以保证全面性，所以测试上并不能严格的发现所有问题，还有可能在一些情况下会出现的奇怪bug可能并没有人能想到从而构造数据。

但是好处是，由于软件是作为黑盒而使用的，因此我们作为使用者只需要关心软件的黑盒性质，与软件具体实现无关，如果软件实现发生了变化，在黑盒测试中花费经历和时间测试用例仍可用。

总而言之，黑盒测试更关注是否能执行要求的任务，关注它对外暴露的功能

白箱测试

和黑盒测试相反，白箱测试更少的关注了程序的外在整体表现，而是更多的关注程序的具体实现。

查阅资料知道，它也称结构测试或逻辑驱动测试，它是知道产品内部工作过程，可通过测试来检测产品内部动作是否按照规格说明书的规定正常进行，按照程序内部的结构测试程序，检验程序中的每条通路是否都有能按预定要求正确工作，而不侧重于关注程序作为一个整体的实现。

了解程序内部逻辑结构、对所有逻辑路径进行测试。“白盒”法是穷举路径测试。在使用这一方案时，测试者必须检查程序的内部结构，从检查程序的逻辑着手，得出测试数据。

一方面，可以由专业的人员来看源码，直接而专业的梳理出每一条程序执行的路径，根据代码的具体实现而分析出程序的问题所在，

我们知道具体要实现什么方法了，所以就针对该方法编写实现它是否满足规格的测试。

好处是，这样做，可以直接x光般透彻的进入代码内部审查，可以明晰的分辨代码的具体问题所在，并由于可以针对每一个内部路径编写相应的测试，覆盖率和检测精确度比黑盒的盲投喂要强。

坏处是，阅读代码分析问题，再针对的编写对板块的测试程序，本身就是个成本极高的工作，譬如我们的OKtest就是很费力写的，同时，笔者帮同学看代码bug和邀请同学审查笔者代码的时候便了解这样的痛苦。

另外，一个程序的内部执行路径数目可能是相当恐怖的，因此白箱测试的代价可能是十分甚至九分的巨大。

其他测试

单元测试

单元测试就是对代码中测试者划定的最小单元为对象测试的方法。

这个单元可以是函数，可以是类，总之是一个有意义的整体。

可以根据这个整体的理想表现，来编写相应的测试程序。

单元测试是建立在白箱的基础上，只有透明了，才能知道单元都是什么。

例如OKtest，就是对一个方法作为单元来编写的测试。

使用这样的测试方法，可以测试每一个模块的执行正确性 ，是否符合规格，是能做到逻辑上正确的方法。

难点在于，编写的时候对每一个单元编写代价较大。

功能测试

功能测试其实和上面说的黑盒测试有类似。

因为投入使用的软件，都必然是对用户不可见的，暴露的只有执行前投入的数据和执行后返回的结果。

因此，只要测试这个软件暴露的结果是否符合设计要求的功能就行了。，确保程序按照预期的方式运行，通过测试系统的所有特性和功能来确保程序与规范的一致即可。

优势和劣势与黑盒类似

集成测试

假设之前说的程序已经测完了，没问题，但是我们是一个大的项目，每一个交出去的程序，都是更大的一部分呢？

那必然有必要把这些程序都整合成整体，然后再次进行测试。

通常，每一个模块，都是一组开发人员进行开发的，由于人的理解和水平差异导致约定和交互接口实现不可能完全精确，很容易出现责任推脱、接口错误、异常不完全等情况。

所以有必要把原本测试好的模块再合并到一起进行测试。

同时，与单元测试相对（每一个模块都根据单元测试进行测试了），在集成测试的时候不必过多关注每一个单元的功能，而更应该关注单元之间通信的正确与否。

和单元测试相结合，有助于一个合格的大型软件。

压力测试

针对卡t的必然要压力狠狠的测试

另外，在定义了一些极限值（比如max）之类的时候，可能在边界条件处理有问题，压力测试就是用极端情况排除这些问题。

好处是，经历压力测试，可以很好的模拟现实生活中的极端情况，从而加强修正或者应对。

例如在课下的时候，把测评机的数据调大，或者专门开一个点qlm，然后极端的测试qlm的性能。

三元环查找

我们需要，找到所有的三角形，而这三角形，我在hash的情况下，先遍历每一个点，但是从a找到的abc，和从b找到的abc就重了（当然有能解决的，这里先打个引子而已）。

有什么办法，能让这个abc的重复，被剪掉，或者说，在找的时候，就不去找？

并查集里面查询连通块的时候，采取了一种方法：给每个集合指定了一个代表元素，这样只需要计算代表元素的个数就行了。

那么对于一个三角形，我们能不能找到一个点，让他，且仅有他，能代表这个 三角形？这样数三角形的时候，只有在从他这里开始数这个三角形才算数。

必须让这个点和另外两个点有不同之处，看在都是点的份上，就是“最小”吧。

我们利用度的概念，来找最小，一个点，连了几个点，就是度数。（度数相同用id）

我们让这个三角形被最小的点代表。

这样，由于id唯一， 所以我们对于每一个三角形，必然能找到一个点，它另外两个点都小，它能代表这个三角形

这样，我们遍历所有点的时候，如果发现，A连接的一个点比A小，那么A一定不能代表这个三角形（如果A和这个点都处在一个三角形里的话）。

用题解的话说，我们给每条边找一个“方向”，让这个方向，满足由小的指向大的。这样，在遍历A周围的点的时候，只需要遍历所有他的出边就行了。

同样的问题下一层遍历的时候还会出现：两条出边，一个和B连，一个和C连，从B算还是从C算？

采取和上面一样的方式，还是选择遍历 出边。比如A找到了出边AB，然后寻找第三个点。我仅仅去遍历B的出边，然后找到的点才算。

这样，从B找到了C，从C的时候便不会找到B。

别忘了，我们始终可以保证A是这个三角形里面最小的。所以，走到这一步，A<B,B<C是可以确定的。

这样，假如A和C之间有边的话，自然这就是一个以A为代表的三角形了。

从B开始遍历的时候，A比B小，也比C小，所以无论如何不会重复。

public int queryTripleSum() {
        if (tripleClean) {
            return tripleSum;
        }
        int count = 0;
        for (Integer integerA : people.keySet()) {
            MyPerson myPersonA = people.get(integerA);
            HashMap<Integer, MyPerson> acquaintanceA = myPersonA.getAcquaintance();
            for (Integer integerB : acquaintanceA.keySet()) {
                MyPerson myPersonB = acquaintanceA.get(integerB);
                if (myPersonA.isLesserThan(myPersonB)) {
                    HashMap<Integer, MyPerson> acquaintanceB = myPersonB.getAcquaintance();
                    for (Integer integerC : acquaintanceB.keySet()) {
                        MyPerson myPersonC = acquaintanceB.get(integerC);
                        if (myPersonB.isLesserThan(myPersonC) && myPersonA.isLinked(myPersonC)) {
                            count++;
                        }
                    }
                }
            }

        }
        tripleClean = true;
        tripleSum = count;
        return count;
    }

显然每个块都有一个代表节点，它的特征是父节点是它自己，只需要遍历所有的点，搜索其中代表节点的数目即可。

public int merge(int id1, int id2) {
        int fa1 = find(id1);
        int fa2 = find(id2);
        if (fa1 == fa2) {
            return -1;
        }
        int rank1 = rank.get(fa1);
        int rank2 = rank.get(fa2);
        if (rank1 < rank2) {
            pre.put(fa1, fa2);
        } else {
            if (rank1 == rank2) {
                rank.put(fa1, rank1 + 1);
            }
            pre.put(fa2, fa1);
        }
        return 0;
    }

按秩合并。中所周知，两个集合合并，只会有一个代表节点，必然从原本的两个代表节点选出，为防止树退化成链表，需要找到深度较小的树，然后合并到深度较为深的树上，（通过让一个代表节点的直接连到另一个的根）这样可以尽量减少树深度的增加。

public int queryBlockSum() {
        if (clean) {
            return blockSum;
        } else {
            clean = true;
            HashSet<Integer> visited = new HashSet<>();
            int count = 0;
            for (Integer id : graph.keySet()) {
                if (!visited.contains(id)) {
                    count++;
                    visited.add(id);
                    LinkedList<Integer> queue = new LinkedList<>();
                    queue.add(id);
                    while (queue.size() != 0) {
                        int theOne = queue.getFirst();
                        queue.removeFirst();
                        for (Integer others : graph.get(theOne).keySet()) {
                            if (!visited.contains(others)) {
                                visited.add(others);
                                queue.add(others);
                            }
                        }
                    }
                }
            }
            blockSum = count;
            clean = true;
            return count;
        }
    }

在图中查询queryblock的时候，采用bfs，设置一个已经访问过的集合。

遍历所有点，然后假设这个点没有被访问过，就对它进行bfs，将所有和它相连的点加入访问集合，同时block数目++。

这样每次进行一个bfs遍历都可以完全的过所有连接的点一遍，就确定了一个连通块。

public boolean isCircle(int id1, int id2) {
        LinkedList<Integer> queueOfOne = new LinkedList<>();
        LinkedList<Integer> queueOfTwo = new LinkedList<>();
        final HashSet<Integer> reachedOne = new HashSet<>();
        final HashSet<Integer> reachedTwo = new HashSet<>();
        queueOfOne.add(id1);
        reachedOne.add(id1);
        queueOfTwo.add(id2);
        reachedTwo.add(id2);
        while (queueOfOne.size() != 0 && queueOfTwo.size() != 0) {
            if (queueOfOne.size() <= queueOfTwo.size()) {
                int head = queueOfOne.getFirst();
                queueOfOne.remove();
                for (Integer point : graph.get(head).keySet()) {
                    if (reachedTwo.contains(point)) {
                        return true;
                    }
                    if (!reachedOne.contains(point)) {
                        queueOfOne.add(point);
                        reachedOne.add(point);
                    }
                }

            } else {
                int head = queueOfTwo.getFirst();
                queueOfTwo.remove();
                for (Integer point : graph.get(head).keySet()) {
                    if (reachedOne.contains(point)) {
                        return true;
                    }
                    if (!reachedTwo.contains(point)) {
                        queueOfTwo.add(point);
                        reachedTwo.add(point);
                    }
                }
            }
        }
        return false;
    }

对于两个点的连通性，选择双向bfs，每次从两个队列里面各拿出来一个点，假设这个点已经被另个一个点访问过，那么自然判定了这两个点连通。

为了防止退化成普通bfs，每次从队列长度较小的那个队列中取点。

public int findMinCircle(int id) {
        HashMap<Integer, Integer> distance = new HashMap<>();
        PriorityQueue<Point> heap = new PriorityQueue<>(POINT_COMPARATOR);
        for (Integer point : graph.keySet()) {
            if (!point.equals(id)) {
                distance.put(point, graph.get(id).getOrDefault(point, Integer.MAX_VALUE));
                heap.add(new Point(point, distance.get(point)));
            }
        }
        HashSet<Integer> reach = new HashSet<>();
        reach.add(id);
        HashMap<Integer, Integer> preTree = new HashMap<>();
        for (Integer side : graph.get(id).keySet()) {
            preTree.put(side, side);
        }
        for (int i = 0; i < graph.size() - 1; i++) {
            int k = 0;
            while (!heap.isEmpty()) {
                Point find = heap.poll();
                if (!reach.contains(find.getId())) {
                    k = find.getId();
                    break;
                }
            }
            if (distance.get(k) == Integer.MAX_VALUE) {
                break;
            }
            reach.add(k);
            for (Integer point : graph.get(k).keySet()) {
                int tmp = graph.get(k).get(point) + distance.get(k);
                if (!reach.contains(point) && distance.get(point) > tmp) {
                    distance.put(point, tmp);
                    heap.add(new Point(point, tmp));
                    preTree.put(point, preTree.get(k));
                }
            }
        }
        int ans = Integer.MAX_VALUE;
        for (Integer point : graph.get(id).keySet()) {
            if (!point.equals(preTree.get(point))) {
                ans = Math.min(ans, graph.get(id).get(point) + distance.get(point));
            }
        }
        for (Integer point1 : graph.keySet()) {
            if (!preTree.containsKey(point1) || point1 == id) {
                continue;
            }
            for (Integer point2 : graph.get(point1).keySet()) {
                if (!preTree.containsKey(point2)) {
                    continue;
                }
                if (point2 != id && !preTree.get(point2).equals(preTree.get(point1))) {
                    ans = Math.min(ans, distance.get(point1) +
                            distance.get(point2) + graph.get(point1).get(point2));
                }
            }
        }
        return ans == Integer.MAX_VALUE ? -1 : ans;
    }
}

求过一个点的最小环长。

采用最短路树的方式，使用堆优化迪杰来找到所有最短路径长度，并存起来每一个点所在的子树（类似并查集，不过代表元素是与源点直接相连的子树）

规格与实现分离

对于规格与实现，我已经在我的帖子(oktest:规格、实现方法和验证方法)里面说明了，在此摘要：

当作为程序员，要实现这个单元的内容，我们应该不会选择严格的按照JML翻译，毕

竟，JML作为客户描述想要的东西究竟满足什么样的特征，为了表达的尽可能严谨，

实现可能会起来效率差且复杂。

我们会选择：阅读JML，理解JML让我们想干什么，然后按照我们自己的理解用自己

的方法去写一个我们认为符合JML想让我们做什么的代码。

比如，我们会进行数据结构的转换。规格给的是[]数组，我们使用<id,Person>的

对于实现目的的手段，我们（作为程序员）做了一个叙述的变换：

我们将JML设想的，用户的需求，变换成了我们（程序员）便于实现的、性能高的叙

述方式，然后按照我们的叙述来写代码。

可能从数据结构上不等价，从算法上不等价。

上面我用并查集求联通而不是找到一个以id1,id2为起始的数组，两个方法可能不等
