OKTEST：规格、实现方法和验证方法

在指导书里面，有这样的一句话：

需要同学们了解OK方法，以及具备为较简单规格写OK测试的能力。

OK测试，到底是在做什么呢？

在这次的作业里面，我们就见到了一个这样的方法，需要我们自己完成。

接下来，我想用客户A和程序员B的视角，分别看一下我们的工作。

客户A：我需要一个queryTripleSum方法，我把我想要的效果，用JML表达出来，交给了我雇佣的程序员。

程序员B：我按照他给的JML，依照自己程序员的经验，完成了一个代码。

现在，作业要求我们，实现一个让客户能够验收的代码，就是下面的OKTEST。

public boolean queryTripleSumOKTest(
    HashMap<Integer, HashMap<Integer, Integer>> beforeData, 
    HashMap<Integer, HashMap<Integer, Integer>> afterData, 
    int result);

为了搞清楚它到底要我们干啥，除了指导书之外，让我们看看官方包源码,这个函数究竟是在什么情况下被调用的：

private void queryTripleSumOKTest() {
        ArrayList<String> cmdList = new ArrayList(Arrays.asList(this.cmds));
        cmdList.remove(0);

        try {
            HashMap<Integer, HashMap<Integer, Integer>> beforeData = this.getModifyRelationOKData(cmdList, true);
            HashMap<Integer, HashMap<Integer, Integer>> afterData = this.getModifyRelationOKData(cmdList, false);
            int result = Integer.parseInt((String)cmdList.remove(0));
            Network tmpNetwork = (Network)this.networkConstructor.newInstance();
            System.out.println(tmpNetwork.queryTripleSumOKTest(beforeData, afterData, result));
        } catch (Exception var6) {
            System.out.println(var6.getMessage());
        }

    }

ArrayList<String> cmdList

所有的东西，都来自于这个输入的string

事实上，输入里面，

qtsok 3 1 2 3 1 2 3 2 1 3 3 4 1 2 4 3 1 2 3 1 2 3 2 1 2 3 4 1 2 4 0

也是一个string。

我们看，其中，result，是作为末尾那个0被传进来的。

同时，int result = Integer.parseInt((String)cmdList.remove(0));

我们并没有给出其他的方法去得到这个result，也就是说，这个result，是要靠外界输入传进来的。

也就是说，你需要对result做的事，就是外面给你一个result传进来，写代码验证看这个result对不对。

（当然，这个仅仅是对result要做的事，剩下的看下面）

规格

注意，客户将自己想要的东西，严格的转化成了JML。

这个就是规格。

在给定JML的情况下，验证这个方法（叫A吧）对不对，我们能不能称它“满足规格” 需要看的是， 这个方法A，是否严格的满足JML里面的每一个条件

但是，对于一个方法，由于这个方法在调用的时候，已经被封装好了，所以我们能从这个方法获得的信息，有且仅有这个方法执行前后程序数据的状态，还有这个方法的返回结果。

事实上，这些东西，也正是代表了这个方法所造成的全部影响。

因此，在官方给定的接口方法中，向我们传递了一个方法作用的环境，和它作用完后环境的结果。

HashMap<Integer, HashMap<Integer, Integer>> beforeData, 
    HashMap<Integer, HashMap<Integer, Integer>> afterData, 
    int result

before，就是这个方法执行前的环境，after，就是这个方法执行后的环境。

result，就是这个方法得到的结果。上面一起构成了该方法造成的影响。

我们需要根据传进来的这些信息，来判断这个方法A的执行，是否严格的满足了JML的要求。

实现方法

当作为程序员，要实现这个单元的内容，我们应该不会选择严格的按照JML翻译，毕竟，JML作为客户描述想要的东西究竟满足什么样的特征，为了表达的尽可能严谨，实现可能会起来效率差且复杂。

我们会选择：阅读JML，理解JML让我们想干什么，然后按照我们自己的理解用自己的方法去写一个我们认为符合JML想让我们做什么的代码。

比如，我们会进行数据结构的转换。规格给的是[]数组，我们使用<id,Person>的hashMap>，建立在我们的经过性能或者逻辑组织的考量上做出的选择。我认为这个能实现[]能实现的。

比如，里面的isCircle方法，我阅读后发现，就是看这两个点是否互相可达，我就会按照并查集的方法去做。在我看来，作为一个代码的提供方，我认为我是完成了JML的要求的。

实现方法，就是我们作为程序员，将JML理解后实现要求所满足的方法。

但是这个真的满足要求吗？

OKTEST：验证方法

事实上，对于实现目的的手段，我们（作为程序员）做了一个叙述的变换：

我们将JML设想的，用户的需求，变换成了我们（程序员）便于实现的、性能高的叙述方式，然后按照我们的叙述来写代码。

当然，这个变换，很有可能不是等价的 。

可能从数据结构上不等价，从算法上不等价。

上面我用hashMap代替了[]，两个数据结构可能不等价。

上面我用并查集求联通而不是找到一个以id1,id2为起始的数组，两个方法可能不等价。

（当然写的正确的话，效果相同，但是直观看来客户会认为不等价，想证明等价，就需要验证）

所以，OKTEST，它作为一个验证方法，就是要看我们（程序员）是不是做出了等价的变换。

现在，回到客户的视角，程序员给你提供了经过变换的一个方法，你看不懂里面是啥，他宣称这个方法符合你的要求，你不信。

但是你可以找出很多种情况，然后让这个方法工作一下，看看它是不是严格符合你想要的结果。

对于这个ok测试就是我们测试这个“严格性”的手段。

作为客户，假设我们要验收程序员给我们的A方法，

我们需要一个检测器（就是OK方法），来检测程序员实现的方法A的正确性。

我们并不观察给出其他人实现的方法A的代码，而是通过给出实现的方法A执行前的状态beforeData和执行后的状态afterData，由我们的ok方法来检验这个方法A执行的正确性。

这里， 我们要做的，就是严格对照JML的要求来看一下是不是满足条件。

譬如：JML说：\result == x

那么我们就要严格的检查，result==x

譬如说：JML说到：

(\exists int i; 0 <= i && i < acquaintance.length; 
      @          acquaintance[i].getId() == person.getId());

那么我们一定就要去在acquaintance里面去找，一定要找到一个acquaintance[i].getId() == person.getId());才算完成目标。

就像这样：

boolean exist_check=false;
for(MyPerson myPerson:acquaintance)
{
	if(myPerson.getId()==person.getId())
	{
	exist_check=true;
	}
}

譬如说：JML说到：

/*@ ensures \result ==
      @         (\sum int i; 0 <= i && i < people.length;
      @             (\sum int j; i < j && j < people.length;
      @                 (\sum int k; j < k && k < people.length
      @                     && getPerson(people[i].getId()).isLinked(getPerson(people[j].getId()))
      @                     && getPerson(people[j].getId()).isLinked(getPerson(people[k].getId()))
      @                     && getPerson(people[k].getId()).isLinked(getPerson(people[i].getId()));
      @                     1)));

那么我们就要用真的去遍历这个三重循环，然后将所有满足上面说的条件的数目加起来，看看 result是不是等于这个值。

注意，我们在做一个合格的OK测试的时候，并不能用任何逻辑和数学的推导来推理出来符合要求，我们必须直观的得到和JML 要求的效果一模一样的效果，这个OKTEST才算是成功的

正如指导书所说：在OKTest中，你需要对JML的全部内容进行检查，除了检验requires和ensures，还有pure、assignable语句等等。例如，对于一个pure方法，调用方法前后的状态应该一致，如果前后状态不一致，那么我们认为这不符合给定的JML

作为一个OKTEST，我们不能依赖任何推导，只接受摆在眼前的结果。

回到题目

回到题目，

                # beforeData
3                 # 有三个人
1 2 3             # 三个人id分别为 1 2 3
1 2 3             # 第1个人有1个熟人，id为2、value为3
2 1 3 3 4         # 第2个人有2个熟人，第1个id为1、value为3，第2个id为3、value为4
1 2 4             # 第3个人有1个熟人，id为2、value为4
                # afterData
3
1 2 3 
1 2 3 
2 1 3 3 4 
1 2 4 
                # result
0

官方的包，提供了这样的一个别人queryTripleNum执行前的环境、执行后的环境、执行的结果，我们需要 严格的验证这些对外暴露的影响，是否满足JML的要求。

在具体实现上，可能有以下几个误区：

1：我们需要用两种不同的 方法算。

错误：这个是让你测别人，不是自己测自己。传进来的result就是 别人执行后的结果。

2：我只需要用自己设计的方法跑一下，结果一样就行了

错误：你的设计和别人的设计一样，都没有说服力，你需要一个严格符合JML的测试方式。

正确的观点：我们只需要用一个方法算一遍，但是不是自己转化后的方法。

就比如说，我自己实现了一个三元环算法的东西来算三角形个数（不符合规格提供的方式，但是符合规格需要的结果），但是为了写这个OKtest，我必须要严格的按规格的过程翻译 一下，而不是拿我自己的方法去比较。

/*@ ensures \result ==
      @         (\sum int i; 0 <= i && i < people.length;
      @             (\sum int j; i < j && j < people.length;
      @                 (\sum int k; j < k && k < people.length
      @                     && getPerson(people[i].getId()).isLinked(getPerson(people[j].getId()))
      @                     && getPerson(people[j].getId()).isLinked(getPerson(people[k].getId()))
      @                     && getPerson(people[k].getId()).isLinked(getPerson(people[i].getId()));
      @                     1)));

一个合格的OKTEST，我们需要严格的把这个翻译成代码。这中间和我自己写的query没任何关系。

/*@ pure @*/

就要我们把pure的含义，直观的翻译成代码去检查，而不是我实现里没改就默认pure了这种。在这个题目里，就是比较这前后两个HASH是否一模一样。

（需要我们实现的OKTEST方法，参数本身是用的hashMap表示。但是从更严格的角度讲，ok方法内部也要像类定义里面一样，构建出规格那样的[]数组)

由于本次的 query 是无副作用的方法，所以有的同学（比如我之前），自己调用一遍自己的 query 然后比对结果。并没有什么意义。

总结

这其实就是强调，规格（JML）是一个很强的可判定的东西
它可以直接转换成ok方法，调用就知道某个具体的“实现代码”是否符合规格
而实现代码只需要满足规格就行，具体实现方法那是另一回事。

OKTEST，做的就是用直观的手段判断只需要满足规格就行这个条件是否满足的方法。
