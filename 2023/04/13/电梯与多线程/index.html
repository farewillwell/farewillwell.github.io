在这次作业中的概览，我们对线程的控制并没有很强的掌控能力，几乎全都是依赖CPU调度的，所谓的“争抢”，我们也不知道结果是啥。

用《Java编程思想》的话来说，就是：因为你对Thread类实际没有任何控制权，（并且这种隔离在使用执行器的时候更加明显，因为执行器将替你处理线程的创建和管理。）

故事背景

为了更好的理解这个问题，我们先假定一个场景，一个只能待一个人的自习室，但是有十名卷王计划在今天进自习室学习。是个卷王就是十个线程(thread)，他们是带着学习的任务(Runnable)来的，大早晨他们来到教室门口(start)—-仿照荣老师的比喻

关于多线程，平凡的obj对象能做到什么？

wait

wait被提取到顶级父类Object对象中，地位等同于toString()方法。

wait的含义是，等待，这个等待，从场景上来讲，是确定暂时放弃这个对象的所有权的等待。

在A线程这个对象调用的wait，那么效果生效：

卷王1号进入教室之后，觉得空气不太好，想出来走一走。所以推门出来。

注意，这个时候，它对教室没有占有权，其他人是可以进去的！（但是只能进去一个人）

  1. 只对当前单个共享变量生效，多个共享变量需要多次调用wait()方法。

  2. 如果线程A调用wait()方法后处于堵塞状态时，其他线程中断（在其他线程调用A.interrupt()方法）A线程，则会抛出InterruptExcption异常而返回并终止

  3. 有两种wait：

    wait(void)代表直接进入等待阶段，直到notify到了才能起来容易死锁，所以注意每次共享资源变换都叫一声。

    wait(int)代表进入等待阶段，过int这么大的时间后假如没被唤醒，函数还是会返回。但是假如这个时候别人拿到锁了，也就会进入阻塞状态。

notify&notifyAll

这两个方法被提取到顶级父类Object对象中，地位等同于toString()方法

Object.notifyAll() 方法用于唤醒在该对象上等待的所有线程（所以搞清楚等的是谁就让谁叫）,而notify则是对调用这个函数的线程进行通知。

这个等待，就是确实的wait()的时候用来激活的。

调用wait()、notify()方法时，当前线程必须要成功获得锁（必须写在同步代码块锁中），否则将抛出异常，要么是在synchronized块里面，要么在synchronized方法里。（其实本质都一样）

什么叫在这个对象上等待的线程？

事实上，这个是wait和notify互相对应的，wait这个东西是哪个对象调用的，也就是说这个进程在等待哪个对象，这个对象抛出的notify也才能解决这个对象上的wait

notifyAll()执行后，只有一个线程能得到锁，其他没有得到锁的线程会继续保持在阻塞状态

wait和sleep

这俩有啥区别？

共同： 都是是这个代码暂时不继续往下执行，在线程上表现为卡住了。

区别：wait没有对象的使用权，但是sleep仍然有使用权。可以理解为一个是卷王在教室里sleep睡觉别人也进不去，但是wait是卷王出来透气其他人就能进去了。

所以说，理论上wait是因为逻辑上条件不够有些事情办不到，所以需要暂时让出来，让其他人先办。

sleep更像一个模拟，就是模拟一件事情需要做的时间，比如卷王睡觉。在这个期间，别人是没有使用权的。

发生主体：wait是任何对象都可以发生，但是sleep的主体是thread类。

你可能写了一个错误，但是idea并没有提示红，但是跑起来直接一个报错：

Exception in thread "Thread-0" java.lang.IllegalMonitorStateException
	at java.lang.Object.notifyAll(Native Method)
	at Producter.run(Productor.java:24)

是因为在非synchronized块里面调用notify了。

简单的来说，当你在调用notify(), notifyAll(),wait(), wait(long), wait(long, int)等线程控制操作方法时，必须要有两个前提。第一：必须要在被synchronized关键字控制的同步代码块中，才能调用这些方法。第二，调用者必须为你当前的锁对象

比如：

private void waitForPerson() throws RuntimeException {
        synchronized (requestTable) {
            try {
                requestTable.wait();
            } catch (InterruptedException e) {
                throw new RuntimeException(e);
            }
        }
    }

这个是电梯线程（Thread）的进入等待模式的方法，在这个块里面调用requestTable的wait，代表调用的电梯线程在这里等request的notify，只有requestTable notify了才会被唤起。

作为调用obj自带的wait函数的，一般都是公共资源，然后在线程里使用obj.wait()（记得用synchronized块包装）

接口与继承，Runnable&thread

关于这个单元，会引入一个叫做多线程的玩意，这个东西在Java里面是自带的东西，所以仅仅需要自己继承就好了。

所以我们需要先了解清楚这Runnable和thread是啥。

优先级

具有较高优先级的线程对程序更重要，并且应该在低优先级的线程之前分配处理器资源。但是，线程优先级不能保证线程执行的顺序，而且非常依赖于平台。

并不能通过设置优先级来确定顺序，只能增加可能性

Thread

Thread类就是代表了线程的抽象，由于线程的启动执行等必然要和底层的操作系统打交道，所有在Thread类中有很多native修饰的本地方法。这个类中也包含了很多对于线程 的操作。比如通过构造方法我们可以创建线程，通过start方法我们可以启动线程，还有interrupt, sleep, join,yield等常用的方法

继承Thread的话，就相当于这些函数都自带好了，想覆写可以覆写。但是不建议

Thread类本身就已经实现了Runnable接口

start

start和直接run有啥区别？

通过该方法启动线程的同时也创建了一个线程，真正实现了多线程。无需等待run()方法中的代码执行完毕，就可以接着执行下面的代码。此时start()的这个线程处于就绪状态，当得到CPU的时间片后就会执行其中的run()方法。这个run()方法包含了要执行的这个线程的内容，run()方法运行结束，此线程也就终止了。

run

在start里面会自动使用run，这样的话run就相当于新开的线程的main函数入口，从这个开始这个线程就脱离原本的main主线程了，开始独立出现了。

但是: 通过run方法启动线程其实就是调用一个类中的方法，当作普通的方法的方式调用。并没有创建一个线程，程序中依旧只有一个主线程，必须等到run()方法里面的代码执行完毕，才会继续执行下面的代码，这样就没有达到写线程的目的.

也就是说，只有start才是新开一个线程的运行。

然后当一个线程死掉，当且仅当run函数跑结束。记得在while(true)里面设计一个break的情况。

sleep

sleep()方法导致了程序暂停执行指定的时间，让出cpu给其他线程，但是他的监控状态依然保持着，当指定的时间到了又会自动恢复运行状态。

sleep并不需要 在synchronized块里面使用，可以直接强行睡眠，我们可以把它用在特定情况的模拟上（比如开关们，上下楼）

在调用sleep()方法的过程中，线程不会释放对象锁，或者说sleep方法对于对象锁本身没啥关系。

也就是说，sleep方法，就是让出了CPU，让这段时间CPU去干别的了，但是对对象的拥有权不会转让。

而且sleep本身要有参数sleep(time)

状态转移

一个线程可以分成4大状态：

  • NEW 新创建

  • RUNNABLE 可运行

  • BLOCKED 阻塞

    –WAITING 无限等待

    –TIMED WAITING 计时等待

  • TERMINATED 终止

刚开始的时候，在执行构造之后，就是新创建状态，然后start，就代表新开了这个线程，进入可运行状态（注意，start并不是代表进入运行状态，运行的状态必须要拿到CPU的时间片，但是刚开始那一下还没有CPU时间片），拿到时间片后，开始跑run，于是就相当于进入运行状态，在运行状态会有三种情况进入阻塞：

  • 阻塞状态:

    可以分为三种：

    • 等待阻塞：运行状态中的线程执行 wait() 方法，使线程进入到等待阻塞状态。
    • 同步阻塞：线程在获取 synchronized 同步锁失败(因为同步锁被其他线程占用)。
    • 其他阻塞：通过调用线程的 sleep() 或 join() 发出了 I/O 请求时，线程就会进入到阻塞状态。当sleep() 状态超时，join() 等待线程终止或超时，或者 I/O 处理完毕，线程重新转入就绪状态。

Runnable

Runnable接口是线程辅助类，仅定义了一个方法run()方法。

但是这个Runnable不能直接作为一个多线程运行，它新开的run也仅仅是当成一个函数运行。

必须通过Thread实例才能创建并运行线程。所以我选择不用Runnable这个玩意

1、定义实现Runnable接口的类，并实现该类中的run（）方法。

2、建立一个Thread对象，并将实现的Runnable接口的类的对象作为参数传入Thread类的构造方法。

3、通过Thread类中的start()方法启动线程，并运行

必须要利用Thread类！！！

然后实际上，单纯的这个是没有sleep方法的，需要在thread里面使用这个sleep方法，借由Thread使用

调用：Thread.sleep()是静态方法，只能控制当前正在运行的线程

共享资源

我们会经常听到：加锁

锁到底是个啥？

谁能进？

用的锁应该是可重入锁，就是调用者可以重新进入，每进入一次锁块（代码块？方法？）都会给这个对象加一层锁，每次退出都会退一层锁。

只要一个对象上的锁数目不是0，它就始终被某个进程掌握

锁什么？

锁对应的东西不是方法也不是属性

锁的是一个对象或者一个类，锁住后，它的所有同步方法都被一个进程掌握，别人不能用，但是其他进程还是能 用它的非同步方法感觉不如全部设成同步的，图个心安

隐式：synchronized

英文：synchronize，同步

synchronized究竟作用于谁？能阻塞啥？

一个有锁的对象，同时只能被一个进程访问同步部分。

调用的时候，加锁，就像给教室交锁一样，其他线程都不能够使用这个教室的同步方法（比如说是擦黑板，就是同步方法，因为几个学生只能有一个擦黑板）但是看黑板，这种其实是不同步的方法，因为你看的时候别人也能看，其他进程还是能访问，有且仅有被synchronized修饰的块不能被访问

（话虽这么说，但是还是所有方法建议全加上锁）

给方法加锁

synchronized void method()
{

}

一旦调用这个方法，就代表**这个对象(this)**而不是类被上锁了，调用这个对象该方法的进程是这个锁的拥有者。在执行完这个方法之前，一直有这个锁存在。

理论上，这个操作比给对象加锁要好。不用在调用的时候给对象专门加锁了，

给对象加锁

synchronized(obj)

{

}

这样的话，在执行这个代码块的时候，obj是被锁住的。在执行结束这个代码块之前，其他线程都不能访问这个对象的synchronized

在锁住的情况下，相对于其他线程，究竟能调用这个被锁住的对象的哪些功能？那又锁住了这个对象的什么？

注意，仅仅是锁了这个对象的syn部分，其他的方法并没有被锁住！！！

这个锁的操作是原子的锁。避免出现一句同步方法，离开后再一句同步方法中间就变化了。

不妨去掉这个类中一些synchronized修饰的方法，经检验，效果是：

线程是交替执行的，说明如果某个线程得到了对象锁，但是另一个线程还是可以访问没有进行同步的方法或者代码。进行了同步的方法（加锁方法）和没有进行同步的方法（普通方法）是互不影响的，一个线程进入了同步方法，得到了对象锁，其他线程还是可以访问那些没有同步的方法（普通方法）。当获取到与对象关联的内置锁时，并不能阻止其他线程访问该对象，当某个线程获得对象的锁之后，只能阻止其他线程获得同一个锁。

简而言之：其他线程对非同步方法的调用不会被锁定 – 任何人都可以同时调用它们。

public synchronized void demo1() {
     while (true) {  //死循环目的是为了让线程一直持有该锁
         System.out.println(Thread.currentThread());
     }
 }
 public void demo1() {
     synchronized (this) {
         while (true) {  //死循环目的是为了让线程一直持有该锁
             System.out.println(Thread.currentThread());
         }
     }
 }

上面说的这两个方法等价。都是在调用这个代码块（方法的本质也是代码块）的时候给对象加上锁，然后其他进程在这个代码块的执行期间均不能访问这个对象的同步方法。

syn锁和wait()开锁之间的关系

synchronized (synchedList) {

           // 列表为空就等待
           while (synchedList.isEmpty()) {
               System.out.println("List is empty...");
               synchedList.wait();
               System.out.println("Waiting...");
           }
           String element = (String) synchedList.remove(0);

           return element;
       }
   }

一个经典的例子就是上面。

遇到空，不能操作，这个时候就选择不去搞，但是按照syn锁的定义，只有走出这串代码块其他线程才能操作这个对象的同步方法。

但是这里如果一直占用着，显然没办法填满。

所以选择用wait暂时开锁，让出控制权，注意这个时候还是在syn块里面，没有退出，这个函数还是没有退出这个块，只不过暂时沉默了，没有执行罢了 但是syn的权利被出让，其他人可以使用。

等其他的一个地方notify的时候（你需要在走出锁的模块之前notifyAll，不然会报错），所有在wait的进程都被唤醒，然后随机竞争这个对象的使用权。

这个时候，被唤醒的进程结束wait状态，然后继续执行那个wait之后的代码，此时由于还是syn，所以立刻恢复掌握权。（假如不是在syn块里面，也没必要wait了，根本就没有拿到锁，你等个啥劲）

显式：lock（特殊情况下用，作业应该用不到）

try&finally子句

关于try…catch的子句，finally的使用：

1、在finally子句中的代码是最后执行的，并且是一定会执行的，即使try里面的代码出现了异常。

2、finally子句必须和try一起出现使用，不能单独编写。
一个try，可以有多个catch，但只能有一个finally，且顺序固定为：try…catch(可以多个)…finally。

一个try只跟一个finally也是允许的，可以没有catch。

一个try，（任意个catch，可以0个），一个finally。

3、finally语句通常使用在哪些情况呢？

当一些代码，不管try里面出现异常，还是没有出现异常，这些代码最终都是必须要执行的，那么这些代码可以放在finally语句中。

如果放在try中，可能这些代码上面就已经出现了异常，而导致这些代码没有执行。

所以说finally语句中的代码一般完成资源的释放与关闭，即使try中出现异常，也能有保障

原子性和同步

如果你正在写一个变量，它可能接下来将被另一个线程读取，或者正在读取一个上一次已经被另一个线程写过的变量，那么你必须使用同步，并且，读写线程都必须用相同的监视器所同步

—–Java编程思想

单独的synchronized方法不足以包装成完整的原子步骤，需要用synchronized块来。

public void run()
   {

但是其实这个代码是不能正常结束的。在消费者这一块：

if(tary.hasCake())
                {
                    System.out.println("eat"+tary.getCake());
                    tary.lostCake();
                }
                tary.notifyAll();
                try {
                    if(!tary.hasCake())
                    {
                        tary.wait();
                    }
                }catch (InterruptedException e)
                {
                    e.printStackTrace();
                }

假如第10个蛋糕生产出来的话，先吃了，然后wait了，wait就是不进入循环了。但是那个生产者已经不生产了，这个时候就无限等待，很呆滞。为了防止，可以多唤醒几次，或者换一下wait的顺讯。

所以关于这种多线程是真的容易bug。

尤其注意，在多线程里面，这些操作的顺序也是需要考虑的。而且有些时候不可复现。因为你不知道谁怎么走的轮次。

notifyall究竟该什么时候唤醒？

比如这种：你不知道while是怎么走的，所以比如说先生产第十个，然后吃了，吃了之后这个循环不走了，去走下一个？

经过 检查，bug是因为：消费者run多run了很长时间，然后从吃完第十个生产的蛋糕后，再次走了一遍循环，这个时候既没有蛋糕生产标志又是TRUE，所以会等，但是下一次跑生产者就直接不生产了，所以就干等着

解决：

synchronized (tary)
            {
                if(hasProducted==10)
                {
                    System.out.println("finish!");
                    tary.moveTary();
                    tary.notifyAll();
                    return;
                }

注意，是在synchronized块里面调用。

\1. 获得当前对象锁的线程，再调用加锁的方法，不会阻塞
\2. 调用该对象普通方法（即没有同步的方法），不会阻塞
\3. 既没有获得锁，调用的又是同步方法或代码块，会被阻塞

运行策略-look

看往届很多都用的look，所以这里省懒劲的话使用look：

我们不妨思考一下平时电梯的运行逻辑，我们想上楼，当我们在进入电梯前，电梯在向上走，于是按下上升键（这个上升键可以由目的层数-当前层数判断，事实上方向也是look的核心之一），发现方向一致，所以电梯来的时候开门了。从电梯上的乘客而言，他们向上走，然后发现到了 一层，然后一个方向和他们相同的人上来了，然后继续往上走。

或者电梯里的人有的要在这里下，所以电梯直接打开，这时候有人在这层的话也直接上（但是一般来说，方向相反的人是不会上电梯的）。

但是假如电梯满员了，笔者天天在上课的时候碰见，就连开门都不开。

假如电梯送走了电梯里的所有人？比如在12楼就没人了。假如上面（16？）有人按电梯，电梯还会继续向上走。否则假如下面有人按电梯，就向下走。但是像都上课的时候，就几乎是一个专门为一个人开的专列。送到了就不动了，所以每次我回宿舍人少的时候就看见电梯停在某些楼，一按按钮才下来。

虽然上面的都是废话，但是很好的解释了look是啥，也解释了look方法的构成方式：假如。。。就。。。

问题是，在什么时候去“假如”？

为了清晰的划分责任，在电梯线程上，需要针对上面的所有操作，每个实现一个具体的方法。move，openAndClose（其实这个中间分开开关还是合到一起见仁见智，可能合到一起出bug的机会少点？），reverse，wait，over。然后在上面那些“假如”发生后，根据假如来判定进行的方法就好了。

我们可以根据对应的操作，把假如抽象成几种情况。

首先大的策略是，只上同向的，能按照原方向走绝对不转向，有目的可走绝对不wait但是这个可能引起耗电量爆炸，这就是调度的事情了）

第一种情况：结束进程：只有电梯里没人，然后请求队里没东西，然后input线程结束了才能-over

第二种情况：开门：上面说过了，里面有人在这楼下||（外面这楼有人在这楼上&&（电梯没满||电梯要下人））

特别的，先下再上，直到装满再走，为了减少开门次数，如果能装的有限，尽量装同一个楼层的。

每一次开门都会完全解决这里这个电梯能解决的问题。

第三种情况：移动：这里不需要解决了但是里面还有人，继续往上走（同一个方向的好处来了）

​ 或者电梯里没有人了，但是在原方向上还有人，继续走。

第四种情况：反向，电梯里没有人了，原方向也没人（原方向优先），反方向有人，反转（这一步只做一个反转，毕竟反正是while）

第五种情况：里面没人外面没请求，wait，（注意是在requestTable.wait())

执行的这几个操作中，电梯处于这几种持续性状态：移动中，等待中，开关门中；有一种状态改变：反向

所以可以看到，在while里面，应该是看看情况->根据情况做动作（一般是持续的，不会出现反复反向）->持续一段状态后看看情况。。。

所以在run的循环里，不用担心轮询，应该是：look->going ->look的流程。（这个going代表是一个状态，在这个状态中，wait是waiting的，那俩都是不在syn中的sleep，都释放了CPU资源）

由此来看看什么时候需要notify？

不妨看看什么情况下需要被notify，也就是什么情况下进的wait，就是一个没人了，所以只要在requestTable里面放人的时候notify一下就好了（当然，为了 保证 终止，一定要在input线程结束的时候notify一下）

这样可以减少很多的notify强制唤醒浪费CPU资源。
