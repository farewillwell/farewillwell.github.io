思考题

thinking1

请根据上述说明，回答问题：在编写的 C 程序中，指针变量中存储的地址
是虚拟地址，还是物理地址？MIPS 汇编程序中 lw 和 sw 使用的是虚拟地址，还是物理地
址？

指针中存储的地址都是虚拟地址，在汇编语言中使用的也是虚拟地址。事实上硬件给程序暴露的都是虚拟地址。

循环链表

理论上插入性能和单向链表相同，但是由于本身存储于实现了一个指向尾部的指针，所以可以直接在尾部插入

双向链表，维护了两个方向的指针，所以插入效率最高，同时由于存储了双向的内容，所以所用空间和每一次修改操作开销较大。但是没有指向尾部的指针，还需要遍历。

thinking6

Thinking 2.6 任选下述二者之一回答：
• 简单了解并叙述 X86 体系结构中的内存管理机制，比较 X86 和 MIPS 在内存管理上
的区别。
• 简单了解并叙述 RISC-V 中的内存管理机制，比较 RISC-V 与 MIPS 在内存管理上
的区别。

选择回答：x86内存管理机制和mips在内存管理上的区别。

x86架构中内存被分为三种形式，分别是逻辑地址（Logical Address），线性地址（Linear Address）和物理地址（Physical Address）

在mips架构中则只有虚拟地址和物理地址。

x86中，通过分段可以将逻辑地址转换为线性地址，而通过分页可以将线性地址转换为物理地址，整体上是段页式的结构。

而mips则是单纯的页式架构。

X86使用的是硬件页表查询，特点是：

1） 操作系统并不知道有TLB存在

2） CPU保证TLB和页表一致性。即当页表改变或者切换时需要刷新TLB。

3） TLB中不存在ASID。

4） OS不能单一操作某一TLB表项，只能通过INVLPG指令无效整个TLB。

MIPS使用的是软件TLB miss方法，使用ASID区分不同的用户空间。具体过程如下：

1） 访问地址TLB不存在时候，陷入内核产生TLB miss异常。

2） TLB miss 处理函数查询三级页表获得地址并填入TLB

3） MIPS架构中访问TLB指令时：tlbwi，tlbri等。

thinkingA1

Thinking A.1 在现代的 64 位系统中，提供了 64 位的字长，但实际上不是 64 位页式存
储系统。假设在 64 位系统中采用三级页表机制，页面大小 4KB。由于 64 位系统中字长为
8B，且页目录也占用一页，因此页目录中有 512 个页目录项，因此每级页表都需要 9 位。
因此在 64 位系统下，总共需要 3 × 9 + 12 = 39 位就可以实现三级页表机制，并不需要 64
位。
现考虑上述 39 位的三级页式存储系统，虚拟地址空间为 512 GB，若三级页表的基地
址为 PTbase，请计算：
• 三级页表页目录的基地址。
• 映射到页目录自身的页目录项（自映射）。

采用自映射的方式，则按照比例换算知，一个页面的大小是4KB，一共2^27个物理页，所以页表共2^27项，来针对所有的页面，然后一页共2^9项，所以三级页表数目为2^18页，为了表示这么多页表，二级页表项2^18个，所以需要2^9二级页表页，所以需要1个一级页表。

其中，已知页表开始的范围是PTbase，所以作为它的目录，开始的点应该是相比于整个空间的比值,PTbase/2^27*2^18=PTbase>>9

是实际上的二级页表的偏移量.然后再根据这二级页表的范围/2^9*1=PTbase>>9>>9，将两次的东西叠加起来，可知。

然后映射到页目录自身的页目录项,则再次偏移即可。

页目录特指一级页表，并不存在第几级页表的页目录。

  1. PTbase+PTbase>>9+PTbase>>18
  2. PTbase+PTbase>>9+PTbase>>18+PTbase>>27

实验难点

理解页控制块和页物理地址的关系

二者没有任何关系

页控制块也不指示页的位置（不论是虚拟还是物理）

页控制块也不放在每一个物理页上。

二者只是一一对应，在虚拟地址中开劈与物理页地址数目相应的页控制块的空间（这个的首地址页和物理地址没任何关系）

然后数组中[i]对应第几个物理页，仅仅是这样的映射，剩下的都没了。

顺便经由此可以理解那几个页的转换宏函数

static inline u_long page2ppn(struct Page *pp) {
	return pp - pages;
}
返回的是当前存的页结构体和pages初始地址的差。由于一一对应，所以其实就是物理的页号
static inline u_long page2pa(struct Page *pp) {
	return page2ppn(pp) << PGSHIFT;
}
注意物理页号是从0开始的，物理页是严格对从0开始的物理地址一片片分成的，所以物理页号<<大小，就是物理地址。
PPN是物理页号的意思。

static inline struct Page *pa2page(u_long pa) {
	if (PPN(pa) >= npage) {
		panic("pa2page called with invalid pa: %x", pa);
	}
	return &pages[PPN(pa)];
}
返回物理地址所在的页，对应的，页管理模块，的地址。

static inline u_long page2kva(struct Page *pp) {
	return KADDR(page2pa(pp));
}
根据页面管理模块的地址来计算页面的物理地址，然后转化成虚拟地址。

对于何时是虚拟地址何时用的是物理地址的理解

页式管理中，最多接触的就是所谓虚拟地址向物理地址的转换，所以在每次写函数的时候容易疑惑哪个是物理地址，哪个是虚拟地址。

比如memsize不是虚拟地址，而是单纯可用物理空间的大小。

freemem也不是一个物理地址，而是指示的是已经使用的虚拟地址的范围。

函数使用的都是虚拟地址。

页控制块的地址也是虚拟地址，内核放的那些0x8000000也是虚拟地址，实际上只有在刚开始探测的时候拿到的才是物理地址（的大小)

其他情况下，虚拟页在调用的时候都被物理地址宏包装起来了。所以几乎仅仅使用了虚拟地址。

对双向链表中pre机制的理解

和之前很不一样的就是在这个的双向链表实现中，指向前一个链表的指针是一个二重指针。

struct type **le_prev; /* address of previous next element */

它的定义也很神奇，指向前一个节点的next指针（而不是前一个节点）

在一开始出了很多问题，后来搞清楚就好了。

pg_walk

最痛苦的就是写pg_walk的时候了

没有梳理明白这个函数究竟是要干什么的，实现什么功能，然后页和项之间的对应关系。

首先：walk并不具备填写表项的功能，就是返回了表项的地址。

我们要做的是：首先给定了一级页表基地址，就是代表从这里开始找偏移了，然后从va这个虚拟地址里面找二级页表项（注意是页表项，不是一个真的页，它的虚拟地址，这个地址必然指了一个页，然后这个页是在二级页表的表项里面存储的，只需要把ppte的值存成这个新建的二级页表就行了

pgdir_walk(Pde *pgdir, u_long va, int create, Pte **ppte)

pgdir是页目录地址，va是虚拟地址，create是控制创造的参量，ppte指向pte的位置。

为啥要用双指针？因为有可能那个pte压根不存在（就是原本以为的二级页表的地址）然后直接把新建的页的页表项的地址（pte*)存在里面就好了。

static int pgdir_walk(Pde *pgdir, u_long va, int create, Pte **ppte) {
	Pde *pgdir_entryp;
	struct Page *pp;
	Pte *pg_phy_entryp;//这个是二级表项的地址。
	/* Step 1: Get the corresponding page directory entry. */
	/* Exercise 2.6: Your code here. (1/3) */
    pgdir_entryp=(Pde*)(pgdir+PDX(va));//

    这个是什么意思？原本一级页表的表头地址pgdir，加上经过PDX(va)（就是地址偏移过后）就是真正这个一级表项的地址。它本身也是个虚拟笛之爱。

	pg_phy_entryp=(Pte*)(KADDR(PTE_ADDR(*pgdir_entryp)));
    然后这个一级表项的地址取值就是表项内容，拿出来取前20位，PTE_ADDr一下就是物理地址，KaDD一下就是虚拟地址，记住这些东西计算的时候都是虚拟地址。这个虚拟地址就是二级页表那一页的表头。

    注意，这个仅在给出的地址有效的情况下能用，无效新开的话还要覆写。
	/* Step 2: If the corresponding page table is not existent (valid) and parameter `create`
	 * is set, create one. Set the permission bits 'PTE_D | PTE_V' for this new page in the
	 * page directory.
	 * If failed to allocate a new page (out of memory), return the error. */
	/* Exercise 2.6: Your code here. (2/3) */
	if((*pgdir_entryp&PTE_V==0))//如果一级表项存的信息是无效的。
	{
		if(create!=0){//假如创造 
			if(page_alloc(&pp)==-E_NO_MEM)
			{
				*ppte=NULL;
				return -E_NO_MEM;
			}没造成直接报错
			else
			{
            pp->pp_ref=1;//注意这个并不代表一个具体的页面地址，它只是一个页管理模块的地址，除了alloc剩下都没法接触真的页物理地址。
			维护新开的页的使用次数
            pg_phy_entryp=KADDR(page2pa(pp))
            然后新开的话，就覆写，让这个成为这个新开的页首地址(pp返回的是这个新物理页的管理模块的地址，虚拟地址，page2pa返回pp这个页管理模块对应物理页的地址)
            (*pgdir_entryp)=(Pde)page2pa(pp);
            然后一级表项内容改成物理地址。
			(*pgdir_entryp)|=PTE_D | PTE_V;
            然后修正特权位。
            造成的话，就相当于申请了一个新的物理页，注意，只是二级页表的物理页，我要的是表项的地址。(而且是虚拟地址)
			}
		}
		else  //不创造的话肯定不够，返回异常就是了。
		{
			*ppte=NULL;
			return -E_NO_MEM ;不创造的话直接报错。
		}
	}
	/* Step 3: Assign the kernel virtual address of the page table entry to '*ppte'. */
	/* Exercise 2.6: Your code here. (3/3) */
	*ppte=pg_phy_entryp+PTX(va);假如一开始有的话虚拟地址偏移之后拿出来二级页表的页内偏移，就恰好是二级表项的地址。
    注意是虚拟地址。

    然后假如新建的话，pg那个已经变成pp了，就是新申请的表的首地址，然后再偏移到这个表项就行了。

实验体会

刚开始看代码看的很崩溃

还是容易对于大量的代码不是很理解，没有完全理解去搜寻文件应该在哪里搜寻（比如h和.c的关系，mmu.h都放了啥之类的）

然后等到在这次的作业中彻底理解并熟悉的文件的分布和其功能时，针对性的查询理解就方便了很多。

然后就是对虚实地址的理解，弯弯绕绕出了很多误区，属于是一开始就有的错误，在给别人讲解的时候才恍然大悟自己错了。

所以千万不要急着写，不要觉得有的代码是自己不要看的，然后凭着自己的理解就开始写，（比如一开始就没搞清楚那几个宏函数是怎么做的，导致后面自己不全函数的时候不知道用哪个）

经过这次实验，自己对于指针的理解和内存模型的理解也提升了很多。
