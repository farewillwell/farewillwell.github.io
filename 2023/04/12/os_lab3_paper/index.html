lab3思考题

BUAA_OS
发布日期:   2023-04-12

Lab3实验报告

思考题

Thinking1

请结合 MOS 中的页目录自映射应用解释代码中 e->env_pgdir[PDX(UVPT)]= PADDR(e->env_pgdir) | PTE_V 的含义。 

#define UVPT (ULIM - PDMAP)
#define PDMAP (4 * 1024 * 1024) // bytes mapped by a page directory entry
#define ULIM 0x80000000

UVPT是ULIM-PDMAP的值，而虚拟地址ULIM是进程和内核地址的分界线从 UVPT 往上到 ULIM 之间则是进程自己的页表。

UTOP = 0x7f400000 ，是为用户所能操纵的地址空间的最大值； ULIM = 0x80000000 ，是操作系统分配给用户地址空间的最大值。两者之间的区域对用户进程而言是一个只读片段，保存着用户页表 UVPT、 struct Page 、 struct Env ，显然这些都是用户程序不允许动的。UTOP 以下的区域就可以被用户程序所读写

宏PDX的定义是这样的 ：

#define PDX(va) ((((u_long)(va)) >> 22) & 0x03FF)

是取这个虚拟地址所对应的页目录编号。

所以env_pgdir[PDX(UVPT)]就是这个地址对应的页目录项的位置（页目录项也是页表项）。

而由于自映射，这个页表项的值，存的应该是这个页目录的物理页号。

所以=PADDR(e->env_pgdir) ，同时要设置为有效，所以或上PTE_V

Thinking2

elf_load_seg 以函数指针的形式，接受外部自定义的回调函数 map_page。请你找到与之相关的 data 这一参数在此处的来源，并思考它的作用。没有这个参数可不可以？为什么？ 

ELF_FOREACH_PHDR_OFF (ph_off, ehdr) {
		Elf32_Phdr *ph = (Elf32_Phdr *)(binary + ph_off);
		if (ph->p_type == PT_LOAD) {
			// 'elf_load_seg' is defined in lib/elfloader.c
			// 'load_icode_mapper' defines the way in which a page in this segment
			// should be mapped.
			panic_on(elf_load_seg(ph, binary + ph->p_offset, load_icode_mapper, e));
		}
	}

在代码中，可以看到

panic_on(elf_load_seg(ph, binary + ph->p_offset, load_icode_mapper, e));

参数是ph，其中ph是每次所针对的段地址的值。

显然不能没有这个参数，如果没有它，就不能指示现在在加载的段是文件中的哪一个段了，也就无从使用了。

Thinking3

结合 elf_load_seg 的参数和实现，考虑该函数需要处理哪些页面加载的情况。

参数有：

Elf32_Phdr *ph, const void *bin, elf_mapper_t map_page, void *data)

分别为段解析结构体地址ph，二进制文件地址bin，回调函数map_page，和进程块地址data。

在使用上，体现为

elf_load_seg(ph, binary + ph->p_offset, load_icode_mapper, e)

然后在函数内部的分支里面，有：

offset != 0
i = offset ? MIN(bin_size, BY2PG - offset) : 0; i < bin_size; i += BY2PG
i < sgsize

这样几个判断。

情况1：若offset！=0先从offset位置开始装载。

情况2：若 BY2PG - offset >= bin_size ，则在上一步中已完成 bin_size 部分的装载，否则需要将 bin_size 循环装载到若干页中，

情况3：装载完必要的部分之后若还没有达到要求的sgsize，那么就继续拷贝页，达到内存空间的分配

Thinking4

• 你认为这里的 env_tf.cp0_epc 存储的是物理地址还是虚拟地址?

虚拟地址

基本上所有都是虚拟地址。

Thinking5

0 号异常 的处理函数为 handle_int，表示中断，由时钟中断、控制台中断等中断造成

1 号异常 的处理函数为 handle_mod，表示存储异常，进行存储操作时该页被标记为只读

2 号异常 的处理函数为 handle_tlb，表示 TLB load 异常

3 号异常 的处理函数为 handle_tlb，表示 TLB store 异常

8 号异常 的处理函数为 handle_sys，表示系统调用，用户进程通过执行 syscall 指令陷入内核

试找出上述 5 个异常处理函数的具体实现位置。

0号：genex.S里面

NESTED(handle_int, TF_SIZE, zero)
	mfc0    t0, CP0_CAUSE
	mfc0    t2, CP0_STATUS
	and     t0, t2
	andi    t1, t0, STATUS_IM4
	bnez    t1, timer_irq
	// TODO: handle other irqs
timer_irq:
	sw      zero, (KSEG1 | DEV_RTC_ADDRESS | DEV_RTC_INTERRUPT_ACK)
	li      a0, 0
	j       schedule
END(handle_int)

12,3,8号：也是genex.S，不过是通过宏实现的

.macro BUILD_HANDLER exception handler
NESTED(handle_\exception, TF_SIZE, zero)
	move    a0, sp
	jal     \handler
	j       ret_from_exception
END(handle_\exception)

以上是定义的宏
BUILD_HANDLER tlb do_tlb_refill

#if !defined(LAB) || LAB >= 4
BUILD_HANDLER mod do_tlb_mod
BUILD_HANDLER sys do_syscall
#endif

BUILD_HANDLER reserved do_reserved

Thinking6

阅读 init.c、kclock.S、env_asm.S 和 genex.S 这几个文件，并尝试说出enable_irq 和 timer_irq 中每行汇编代码的作用。

注释随行解释。

LEAF(enable_irq)//是一个叶子函数，不再调用子函数
	li      t0, (STATUS_CU0 | STATUS_IM4 | STATUS_IEc)//后面定义的是一些各个位的宏，用t0存起来
	mtc0    t0, CP0_STATUS//把上面那些值存到cp0的status里面，相当于启用这些中断。
	jr      ra//转到ra存的地址，相当于返回
END(enable_irq)//结束定义
timer_irq://时钟中断
	sw      zero, (KSEG1 | DEV_RTC_ADDRESS | DEV_RTC_INTERRUPT_ACK)//通过向后面的地址写0，来实现相应时钟中断。其中后面的地址是gexmul模拟出来的地址。
	li      a0, 0//给参数传成0
	j       schedule//跳转到schedule函数

其中，schedule是在shed.c的这个函数

void schedule(int yield) {

事实上，handle_int 函数的最后跳转到了 schedule 函数，而由于调度使用的是时间片轮转，所以时间片正好在被用完后调用这个中断，利用schedule重新规划。

Thinking7

阅读相关代码，思考操作系统是怎么根据时钟中断切换进程的。

事实上，是在时钟中断的timer_irq里面调用的schedule函数,也就是说，每次调用时钟中断，就会依据情况判断执行schedule。

具体执行的进程切换，则采取schedule的函数：

if(yield!=0||count==0||e==NULL||e->env_status==ENV_NOT_RUNNABLE)
	 {
		if(e!=NULL&&e->env_status==ENV_RUNNABLE)
		{
			TAILQ_INSERT_TAIL(&env_sched_list,(e),env_sched_link);
		}
		if(TAILQ_EMPTY(&env_sched_list))
		{
			panic("no runnable envs");
		}
		e=TAILQ_FIRST(&env_sched_list);
		count=e->env_pri;
		TAILQ_REMOVE(&env_sched_list,(e),env_sched_link);
	 }
	 count--;
	 env_run(e);

根据当前的count等等来判断是否需要轮换。

其中，由于a0=0（上面代码的设置），yeild=0，默认仅仅查看count和e的状态。

难点分析

本次课下难度代码填写难度较低，主要在于最后一个空：schedule的实现。

省流：以下内容都是在阐明笔者如何写出来schedule的

因为其他的几乎都是按照上下文填写1行代码即可，非常的自然，但是schedule却需要自己依靠去写整个一串代码，十分的痛苦。

因此本部分主要着重在：我是怎么写schedule的？

在代码里，给出了这样的提示：

static int count = 0; // remaining time slices of current env
	struct Env *e = curenv;

	/* We always decrease the 'count' by 1.
	 *
	 * If 'yield' is set, or 'count' has been decreased to 0, or 'e' (previous 'curenv') is
	 * 'NULL', or 'e' is not runnable, then we pick up a new env from 'env_sched_list' (list of
	 * all runnable envs), set 'count' to its priority, and schedule it with 'env_run'. **Panic
	 * if that list is empty**.
	 *
	 * (Note that if 'e' is still a runnable env, we should move it to the tail of
	 * 'env_sched_list' before picking up another env from its head, or we will schedule the
	 * head env repeatedly.)
	 *
	 * Otherwise, we simply schedule 'e' again.
	 *
	 * You may want to use macros below:
	 *   'TAILQ_FIRST', 'TAILQ_REMOVE', 'TAILQ_INSERT_TAIL'

首先第一行可以看到，count=0；它是一个static，也就是说这个东西是不仅仅在函数中生效的，它一直全局生效，而写在这个函数里就代表仅仅有这个函数操作这个变量。

接下来是一句：

We always decrease the 'count' by 1.

不得不承认这个会让我想先count–，但是显然不行，因为我注意到，后面有一个

or 'e' (previous 'curenv') is* 'NULL'，set 'count' to its priority

这个属于一种很有意思的写法，就是自初始化，由于对这一变量初始化需要的步骤和后面的函数几乎一样，所以不去专门设置一个函数初始化，而在这个函数运作的时候判断是不是第一次使用这个函数。剩下的功能也能保证可以恰当的初始化。

之前没有初始化的时候，cur_env就是NULL，也就是说这个兼具count初始化的功能。所以刚开始不能–了，要等确认或分配后再–。

接下来就是确认该干啥：

If 'yield' is set, or 'count' has been decreased to 0, or 'e' (previous 'curenv') is
	 * 'NULL', or 'e' is not runnable, then we pick up a new env from 'env_sched_list' (list of
	 * all runnable envs)

直接翻译即可，注意这些情况仅仅是需要操作的并列，接下来要对每一个情况分别搞一下。

if(yield!=0||count==0||e==NULL||e->env_status==ENV_NOT_RUNNABLE)
set 'count' to its priority   schedule it with 'env_run'. **Panic
	 * if that list is empty

注意，在这个里面，要分别考虑panic，一旦是空的，就panic

if(TAILQ_EMPTY(&env_sched_list))
		{
			panic("no runnable envs");
		}

然后假如这个是原本可以执行，不过当前根据优先级分配的机会已经用完了，就是count=0.那么：

if(e!=NULL&&e->env_status==ENV_RUNNABLE)
		{
			TAILQ_INSERT_TAIL(&env_sched_list,(e),env_sched_link);
		}

插到最尾部。

这样，无论e是否为NULL,我们都已经让当前的e没意义了，所以直接从新拿取，设置时间片：

e=TAILQ_FIRST(&env_sched_list);
		count=e->env_pri;
		TAILQ_REMOVE(&env_sched_list,(e),env_sched_link);

采用每分配一次机会前就count–的方式，用count指示用完本次后剩余的机会，然后运动即可。

count--;
	 env_run(e);
